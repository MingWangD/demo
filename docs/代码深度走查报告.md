## 4. 发现的技术风险与改进建议（不影响当前答辩，但建议迭代）

### 4.1 学生端“越权读取/修改”风险（优先级：高）
当前 `StudentController` 多个接口仍接受 `studentId` 请求参数并直接下传服务层，未强制等于 `AuthContext.userId()`。理论上学生可在同角色下访问他人ID数据。

**建议**：
- 学生端接口统一去掉 `studentId` 入参；
- 由后端强制使用 `AuthContext.userId()`；
- 或在服务层增加 `if (!studentId.equals(AuthContext.userId())) throw ...`。

### 4.2 接口权限粒度依赖 URI 前缀（优先级：中）
`JwtInterceptor` 目前用 URI 前缀做权限判断，维护成本低但可读性与可扩展性一般。

**建议**：
- 中期引入注解化权限（如方法级角色注解）或 Spring Security；
- 保留 `AuthContext` 作为业务身份上下文。

### 4.3 HTTP 401 场景返回状态码为 200（优先级：中）
拦截器中 `write401` 将 HTTP 状态设为 200，仅业务 code=401。前后端对接可用，但不利于标准化网关/监控统计。

**建议**：
- 返回真实 HTTP 401；
- 响应体仍可保留现有 `Result` 结构。

### 4.4 特征提取中 `attendanceRequired` 目前为固定默认值（优先级：中）
`FeatureExtractor` 内部 `attendanceRequired` 默认为 10，未按课程读取 `attendance_required_count`。

**建议**：
- 若 `courseId != null`，读取课程维度阈值并计算 `attendanceRate`，保持与考试资格规则一致。

### 4.5 预测与预警数据可能增长较快（优先级：低）
每次触发重算会新增 `student_feature/risk_prediction/warning_record`，长期运行会膨胀。

**建议**：
- 增加“同日同学生同课程只保留最新”策略或归档任务；
- 看板读取处可显式按时间排序+分页。
