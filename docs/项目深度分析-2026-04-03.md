# 项目深度分析（2026-04-03）

## 1. 总体定位与业务闭环
项目定位为“高校学情智能预警系统”，业务链路覆盖：
登录/出勤采集 → 作业/考试行为沉淀 → 特征提取 → 逻辑回归风险预测 → 预警落库 → 教师干预。

该定位在 README 中有完整叙述，并明确了学生端和教师端双角色场景。

## 2. 技术架构拆解

### 2.1 前后端技术栈
- 前端：Vue3 + Vite + Element Plus + Axios + Vue Router + ECharts。
- 后端：Spring Boot + MyBatis + MySQL + Hutool JWT。

### 2.2 分层架构（后端）
- Controller：鉴权后暴露 REST 接口。
- Service：业务编排与权限校验。
- Mapper + XML：持久层 SQL。
- Algorithm：`FeatureExtractor` + `LogisticRegression` + `RiskPredictor` 组成核心算法链。

## 3. 数据模型与可扩展性
数据库模型设计完整，覆盖：
- 用户与课程：`sys_user`、`course`、`student_course`
- 学习过程：`student_attendance`、`homework`、`homework_submission`、`exam`、`exam_record`、`exam_qualification`
- 学情风险：`student_academic`、`student_feature`、`risk_prediction`、`warning_record`、`intervention_record`

该设计具备良好的“追溯性”：可以回放每次预测的输入特征与预测结果，便于答辩讲解与后续模型评估。

## 4. 风险引擎工作流

### 4.1 触发机制
- 学生登录时：自动补录当日出勤并触发每门课的风险重算。
- 定时任务：每天凌晨 2 点全量重算。
- 业务动作：作业/考试相关流程中也会触发重算（由各服务调用）。

### 4.2 核心计算链
1. `FeatureExtractor` 汇总并标准化特征（出勤率、作业提交率、作业均分、考试均分、GPA、缺交缺考等）。
2. `LogisticRegression` 输出风险概率。
3. `RiskPredictor` 按阈值转成 HIGH/MEDIUM/LOW，并生成可解释“主原因”。
4. 写入 `risk_prediction` 与 `warning_record`，形成预警闭环数据底座。

## 5. 鉴权与权限控制评估

### 5.1 现有优点
- 全局拦截器统一校验 JWT，避免 Controller 重复鉴权代码。
- 支持按路径前缀做角色约束：`/api/student/**`、`/api/teacher/**`。
- 关键教师操作在 Service 层做“课程归属校验”，防止越权。

### 5.2 风险点
- JWT 密钥硬编码在代码中，不利于生产环境密钥治理。
- token 校验失败统一返回 HTTP 200 + 业务码 401，会影响网关、监控、前端拦截的标准语义。
- 密码对比为明文比对，缺少加盐哈希。

## 6. 前端实现评估

### 6.1 结构优点
- 路由按角色页面分组清晰（学生端、教师端）。
- Axios 请求拦截器统一附带 token，响应拦截器统一处理 401 跳转登录。
- 教师端围绕“看板-筛选-详情-干预”完整串联业务闭环。

### 6.2 可改进点
- 缺少前端路由守卫（beforeEach）做“未登录访问控制”和“角色路由控制”，主要依赖后端拒绝。
- 状态存储依赖 localStorage，建议引入 Pinia 统一会话状态管理。

## 7. 代码质量与潜在问题

1. `TeacherService` 中 `ExamMapper` 重复注入声明，存在明显冗余，应去重。  
2. `FeatureExtractor` 存在未使用变量 `Course course = null;` 和注释遗留，提示该段曾计划使用课程参数但未落地。  
3. `StudentController#updateAcademic` 使用传入 `studentId` 直接更新，没有通过 `resolveStudentId` 绑定当前学生身份，存在学生越权修改他人学业信息风险。  
4. 全局异常处理对 `Exception` 统一返回笼统错误，缺乏错误码分层与 traceId，不利于线上排障。  

## 8. 性能与数据一致性观察
- 当前有多处“全表查后 Java 侧过滤”写法（尤其在高风险筛选、作业提交统计等），数据量放大后将导致内存和响应时间压力。
- 预测结果按“插入新记录”保存而非覆盖更新，适合做时间序列分析，但需要配套归档策略避免表膨胀。
- 定时全量预测与业务触发预测并存，需关注并发写入时的重复预警记录问题。

## 9. 安全与工程化建议（优先级）

### P0（必须尽快）
- 密码改为 BCrypt（或 Argon2）并迁移历史用户口令。
- JWT 密钥移入环境变量/配置中心，支持轮换。
- `updateAcademic` 强制绑定当前登录学生，或仅允许教师端操作。

### P1（建议近期）
- 将 401/403 改为标准 HTTP 状态码。
- 将高风险、作业统计等查询下推到 SQL（分页 + 条件过滤 + 索引）。
- 为 `risk_prediction(student_id, course_id, predict_time)` 等高频字段补充复合索引。

### P2（中期优化）
- 引入统一审计日志与链路追踪（traceId）。
- 把算法参数（权重、阈值）配置化，支持灰度调整。
- 增加自动化测试：鉴权、权限边界、关键业务流程回归。

## 10. 结论
这是一个“业务闭环完整、答辩展示友好、工程上具备继续演进空间”的项目：
- **强项**：端到端流程完整、数据模型清晰、风险解释性较好。
- **短板**：安全基线与查询性能仍偏“演示版”思路。
- **建议**：优先补齐安全与权限漏洞，再优化 SQL/索引与可观测性，即可快速迈向可上线试运行版本。

## 11. 面向“毕业设计演示”的最小修改清单（不追求生产级）
如果目标只是“答辩现场稳定演示”，建议只做下面 6 件事：

1. **补一个前端路由守卫**：未登录跳 `login`，防止老师现场点链接直接白屏或越权页面。  
2. **修复 `updateAcademic` 越权点**：学生只能改自己的数据（或直接只允许教师调用）。  
3. **去掉重复注入与明显冗余**：`TeacherService` 的重复 `ExamMapper` 字段删掉，避免被问到代码规范问题。  
4. **把 JWT 密钥移到配置文件**：哪怕只放 `application.yml` 也比硬编码好，答辩时更好解释“可配置化”。  
5. **高风险列表加分页（哪怕假分页）**：避免数据稍多时页面卡顿，演示更流畅。  
6. **准备一套“固定演示数据”**：保证高风险、干预、趋势图都有可展示结果，降低翻车概率。  

### 建议的演示前自测（15 分钟）
- 教师登录 → 看板加载正常；
- 高风险筛选（按课程+风险等级）能查到数据；
- 学生详情能看到风险原因、成绩、出勤；
- 学生端提交一次作业后，刷新能看到风险趋势变化；
- 干预记录可新增并能回显。

> 以上改动足够支撑“可讲清楚 + 可跑通 + 不容易现场报错”的毕业设计演示目标。
